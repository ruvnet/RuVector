## TL;DR — Mixed Real-Data Results

v2 established that `LogQuantized + UnifiedDistanceParams` (SIMD-accelerated) is a clean win on **synthetic** datasets. v3 runs the **exact same benchmark** on standard public ANN datasets: SIFT1M and GloVe.

**The verdict is nuanced and data-dependent:**

| Dataset | Headline finding |
|---|---|
| **SIFT1M (Euclidean, 128D, 100K subset)** | ✅ **+14.0% QPS at ef=64**, Recall@1 +0.75%, build time −3.3%, p50/p95/p99 at ef=64 all better. Mixed at higher ef_search: p99/p99.9 ef=256 regress 7.7%/36.9%. |
| **GloVe-100d (Cosine, 100K subset)** | ⚠️ **−10.4% QPS at ef=256**, p95/p99 regress 10–12%. Recall preserved. Wins only at tail p99 ef=64. |

**The v2 synthetic wins did not fully transfer to real data.** On SIFT1M the benefit is real and meaningful at low ef_search; on GloVe (Cosine, 100D, non-power-of-2 dim) the benefit inverts at high ef_search. Recall is preserved on both.

## Why the Real-Data Result Diverges from Synthetic (Honest Analysis)

v2 (synthetic Normal embeddings) showed +11.5% QPS at ef=64 and +8.5% at ef=256 — a clean, consistent win. Real GloVe shows a clear regression at ef=256. Hypotheses in priority order:

1. **Dimension alignment (most likely)**: GloVe is 100-dimensional. SimSIMD and the scalar tail handling in `UnifiedDistanceParams::compute()` are both optimized for power-of-two vector widths (SIFT=128, synthetic=128). At 100D the tail remainder loop executes every call, and the two code paths (baseline vs unified cascade) may handle the tail with different efficiency. This would explain why SIFT1M (128D) wins while GloVe (100D) regresses.

2. **Cosine branching differences**: In `UnifiedDistanceParams::compute()`, Cosine dispatches to `simsimd::SpatialSimilarity::cosine()` directly, which is the exact same call the baseline `distance()` function makes. So the theoretical branch-elimination advantage is zero for Cosine — both paths call the same SimSIMD function. Any difference is pure overhead: the `if self.normalize` check vs the `match metric` dispatch. Mild compiler codegen differences could explain either direction.

3. **Data structure in GloVe**: Word embeddings have different spatial structure than SIFT (which is image descriptors). HNSW graph topology during construction may interact differently with distance-function characteristics.

4. **Cache/memory effects**: GloVe vectors are 400 bytes (100 × 4), SIFT are 512 bytes (128 × 4). Cache-line alignment and prefetching may differ — though this would affect both paths equally.

**Most likely: (1) + (2) combined.** The branch-free claim is only load-bearing for non-normalized metrics (Euclidean, DotProduct, Manhattan). For Cosine the two paths are functionally identical SimSIMD calls behind different conditionals, so net benefit is near zero before any noise.

## Hybrid Score Fusion Micro-Benchmark (EmlScoreFusion)

Since `EmlScoreFusion` remains in the PR as an opt-in hybrid-search primitive, I added a micro-benchmark comparing its wall-time overhead vs linear fusion on 10K synthetic (vector_score, bm25_score) pairs:

```bash
cargo bench -p ruvector-core --bench eml_end_to_end -- eml_e2e_score_fusion
```

| Benchmark | Mean time (10K pairs) | Per-pair |
|---|---|---|
| `linear_fusion_10k` | 7.66 µs | 0.77 ns/pair |
| `eml_fusion_10k` | 32.79 µs | 3.28 ns/pair |
| **Overhead** | **+4.3× wall time** | **+2.5 ns/pair absolute** |

The **semantic** claim — that EML fusion captures a 3.16× vector/keyword asymmetry — is validated by `tests/eml_proof.rs` (PROOF 5). The wall-time benchmark above measures the **cost** of that asymmetry vs the cheaper linear fusion: 4.3× slower in relative terms, 2.5 ns per pair in absolute terms. For typical hybrid-search workloads (top-K fused per query, K in the tens to hundreds), the overhead is sub-microsecond and invisible to end-users. For batch scoring of millions of pairs, prefer linear fusion unless corpus evidence shows the asymmetry matters for retrieval quality.

## Final Recommendation

Given real-data evidence, the merge strategy proposed in the preceding report needs revision:

### Ship as opt-in, with clear guidance

1. **`UnifiedDistanceParams::compute()`**: keep the SIMD port; it's correct and tests pass. But document that it's **not a universal QPS win** — its benefit is metric- and dimension-dependent. Recommend only for Euclidean at power-of-two dimensions based on current evidence.

2. **`HnswIndex::new_unified()`**: keep the opt-in constructor. Default `HnswIndex::new()` stays dispatched.

3. **`QuantizationConfig::Log`**: keep as opt-in. Doc comment now explicitly states:
   > LogQuantized shines on heavy-tailed distributions (exponential, ReLU outputs, frequency data). On normal/SIFT-like embeddings it has higher reconstruction MSE but does not degrade recall.

4. **`EmlTree` / `EmlModel`**: remain in the codebase under `advanced::eml` for the learned-index use-case (converges on `exp(x)` exactly — genuinely novel). No product impact; not wired into default search path.

5. **`EmlScoreFusion`**: keep as opt-in. Worth it for asymmetric corpora; linear fusion remains default.

### Do NOT set any EML feature as default

The real-data evidence is mixed enough that silently defaulting would hurt ~half of users. Making everything explicit opt-in (which is already the case) is the safe ship.

### Next investigative step (recommended before merging as opt-in)

Verify the dimension-alignment hypothesis: rerun GloVe padded to 128D (zero-pad the last 28 dims) and see if the regression disappears. If it does, the correct action is either:
- Add a dim-alignment helper in `UnifiedDistanceParams` that pads to a multiple of 8 behind the scenes, or
- Document the caveat explicitly ("expect wins only on power-of-two dimensions").

## Reproducibility

```bash
# Pre-cache datasets (one-time, ~1GB)
mkdir -p bench_data && cd bench_data
curl -fLO ftp://ftp.irisa.fr/local/texmex/corpus/sift.tar.gz && tar xzf sift.tar.gz
curl -fLO https://nlp.stanford.edu/data/glove.6B.zip && unzip -o glove.6B.zip glove.6B.100d.txt
cd ..

# Run the full real-data proof (takes ~15 min on M-series Mac at 100K subset)
EML_FULL_PROOF=1 EML_REAL_DATASETS=1 EML_SYNTHETIC_DATASETS=0 \
  EML_PROOF_N=100000 EML_PROOF_Q=500 \
  cargo bench -p ruvector-core --bench eml_end_to_end -- eml_e2e_full_proof

# Add synthetic for comparison (omit EML_SYNTHETIC_DATASETS=0)

  cargo bench -p ruvector-core --bench eml_end_to_end -- eml_e2e_full_proof

# Full 1M SIFT1M (takes 10-30x longer — ~3+ hours)
EML_FULL_PROOF=1 EML_REAL_DATASETS=1 EML_PROOF_N=1000000 \
  cargo bench -p ruvector-core --bench eml_end_to_end -- eml_e2e_full_proof
```
