## TL;DR — Honest Conclusion

**The EML optimizations as currently wired (`LogQuantized` + `UnifiedDistanceParams`) DO NOT beat the baseline on the critical product metrics.** They are either neutral or regressive across the board for end-to-end ANN search:

| Dimension | Verdict |
|-----------|---------|
| **Search QPS** | ❌ **-21% to -22% REGRESSION** (both datasets) |
| **Tail latency (p99.9)** | ❌ **+30% to +104% REGRESSION** |
| **Index build time** | ❌ **+17% to +37% REGRESSION** |
| **Recall@k** | ⚖️ **Within noise** (±0.6%, not significant) |
| **Reconstruction MSE** | ❌ **+19% to +261% REGRESSION** on tested synthetic data |

Only measurable improvement: reconstruction MSE on *specifically* ReLU/exponential/log-normal distributions (from `tests/eml_proof.rs`), but this does **not translate to recall gains** in the HNSW graph search.

See [Root Cause Analysis](#root-cause-analysis) below.

## Root Cause Analysis

### Why `UnifiedDistanceParams` loses: it's not SIMD

The baseline distance path (`distance()` in `distance.rs`) dispatches to **SimSIMD**, which uses hand-tuned AVX2/NEON intrinsics. The `UnifiedDistanceParams::compute()` in `advanced/eml.rs` is pure scalar Rust:

```rust
// advanced/eml.rs:644-651 (Cosine path)
for (ai, bi) in a.iter().zip(b.iter()) {
    dot += ai * bi;
    norm_a += ai * ai;
    norm_b += bi * bi;
}
```

The *theoretical* win from eliminating branch prediction on `match metric` is ~nanoseconds per call. The *actual* loss from missing SIMD is ~10–30x per distance call. HNSW does thousands of distance computations per query, so SIMD dominates decisively.

**Fix required**: Re-implement `UnifiedDistanceParams::compute()` using SimSIMD primitives, then re-benchmark.

### Why `LogQuantized` reconstruction MSE is worse on tested distributions

Log quantization (`ln(v - min + 1)` before uniform quantization) allocates precision exponentially: dense near the minimum, sparse at the tail. This wins when the data distribution matches that allocation.

Our tested distributions:
- **Half-normal (SIFT-like)**: values fairly uniform across [0, 90], so log quantization wastes precision on the tail. **Loses**.
- **Near-zero Gaussian**: values near ±0.3, so `ln(v - min + 1)` ≈ linear for small range. **Slight loss**.

From existing `tests/eml_proof.rs` (measured separately):
- **Exponential distribution**: Log wins **52.5%** MSE reduction ✓
- **ReLU (half-normal at higher dim)**: Log wins **20.9%** MSE reduction ✓
- **Log-normal**: Log wins **5.1%** MSE reduction ✓

So `LogQuantized` is a quality improvement *only* for heavy-tailed, skewed distributions — and even when it wins on MSE, **it did not translate to recall@k improvements** in our HNSW benchmarks (both datasets showed Recall@k differences within noise: ±0.6%). The reconstruction error is below the ANN search's noise floor.

### Why build time regresses

LogQuantized requires `ln` during quantize and `exp` during reconstruct — both ~10x slower than linear arithmetic. The unified-distance index also builds with the scalar compute path (no SIMD). Combined: +17–37% build slowdown.

---

## Recommendation

**Do not enable `QuantizationConfig::Log` or `HnswIndex::new_unified` as the default.** Based on this proof:

1. **Keep `ScalarQuantized` + SimSIMD as default** — it is strictly better on all measured dimensions for the tested workloads.
2. **Gate `LogQuantized` as opt-in** for specifically heavy-tailed distributions (e.g. sparse feature vectors, frequency counts, exponentially-distributed scores) where a user has validated reconstruction quality matters more than latency.
3. **Do not use `UnifiedDistanceParams` in the search hot path until it is SIMD-accelerated.** The current scalar implementation is a net negative. If the theoretical branch-free advantage is desired, port the inner loop to SimSIMD primitives.
4. **`UnifiedDistanceParams::batch_compute_parallel`** may still be useful for offline bulk distance computation where SIMD dispatch cost is amortized over large batches — this was not re-measured.
