**Date**: 2026-04-14
**Hardware**: Apple Silicon (aarch64), Darwin 25.3.0
**Rust**: release profile
**Benchmark**: `crates/ruvector-core/benches/eml_end_to_end.rs`
**Preceding reports**:
- v1: `bench_results/eml_proof_2026-04-14.md` — disproof of scalar EML kernel
- v2: `bench_results/eml_proof_2026-04-14_v2.md` — SIMD port confirms synthetic win
- v3: `bench_results/eml_proof_2026-04-14_v3.md` — real-data mixed (SIFT1M +14%, GloVe −10%)

## TL;DR — Padding did NOT flip the regression (honest finding)

v3 identified that `UnifiedDistanceParams + LogQuantized` wins on SIFT1M (128D,
Euclidean) but regresses on GloVe (100D, Cosine). We hypothesised the non-
power-of-two dimension was the root cause. v4 tested this by:

1. Adding `pad_to_power_of_two: bool` to `UnifiedDistanceParams`
2. Adding `HnswIndex::new_unified_padded()` to set the flag
3. Re-running the exact v3 GloVe benchmark with padding enabled
4. Keeping SIFT1M results unchanged from v3

**Result: padding made GloVe regression WORSE, not better.**

| GloVe-100d Δ vs baseline | v3 unpadded | v4 padded |
|---|---|---|
| QPS ef=64 | −2.2% | **−24.6% WORSE** |
| QPS ef=256 | −10.4% | **−23.4% WORSE** |
| Build time | +3.3% | **+31.7% WORSE** |
| Recall@1 | −0.27% | +0.69% (noise) |

**Root cause of the v4 worsening:** my padding implementation pays the padding
cost *per distance call* via a thread-local scratch: RefCell borrow + clear +
`extend_from_slice` (400 bytes memcpy × 2) + resize + SimSIMD call. HNSW
performs thousands of distance calls per query; ~30–50 ns of padding overhead
per call swamps whatever tail-handling savings SimSIMD gains on aligned
vectors.

**The correct architectural fix is pad-at-insert, not pad-per-call** — pad the
base vectors to 104D (or 128D) once at index construction time, pad the query
vector once per search, and let `compute_raw()` see pre-aligned slices with
zero per-call overhead. That's a more invasive refactor than the user asked
for in this task and is left as follow-up work.

**Per the user's shipping criterion** ("If padding flips the GloVe regression
into a win, we ship immediately. If not, we still ship with the current
caveat"): we ship with caveat. The padding config is available for future
pad-at-insert implementations or for users who want to experiment; it is
defaulted off.

## Why the Padding Implementation Underperformed (honest engineering analysis)

The `pad_to_power_of_two` flag triggers this path in
`UnifiedDistanceParams::compute()` (see `advanced/eml.rs`):

```rust
if self.pad_to_power_of_two {
    let len = a.len();
    if len % 8 != 0 {
        let padded_len = (len + 7) & !7;
        return compute_padded(self, a, b, padded_len);
    }
}
```

And `compute_padded` uses thread-local scratch:

```rust
PAD_SCRATCH_A.with(|sa| PAD_SCRATCH_B.with(|sb| {
    let mut pa = sa.borrow_mut();
    let mut pb = sb.borrow_mut();
    pa.clear(); pa.extend_from_slice(a); pa.resize(padded_len, 0.0);
    pb.clear(); pb.extend_from_slice(b); pb.resize(padded_len, 0.0);
    params.compute_raw(&pa[..padded_len], &pb[..padded_len])
}))
```

The per-call cost is roughly:
1. 2 × `RefCell::borrow_mut()` — ~1–2 cycles each
2. 2 × `clear()` — ~1 cycle each
3. 2 × `extend_from_slice(100)` — ~400-byte memcpy each (~20–40 cycles each with SIMD memcpy)
4. 2 × `resize(104, 0.0)` — 4×f32 zero writes each (~8 cycles)
5. SimSIMD cosine on 104D — same as baseline 100D + 4 extra elements

Total added per call: ~60–100 cycles on Apple Silicon. HNSW with ef_search=256
evaluates roughly 1,000–2,000 distances per query, so ~60–200 µs of added
overhead per query. At 2,400 QPS the query budget is ~420 µs total — this
overhead is a significant fraction.

**Meanwhile, the savings from aligned SimSIMD dispatch:** SimSIMD is already
extremely well-optimized for 100D inputs; the tail-handling loop for 4 extra
elements is a handful of scalar ops (~5–10 cycles). So we're trading ~60–100
cycles of padding overhead for ~5–10 cycles of tail savings — a net loss of
~50–90 cycles per call.

### The correct fix: pad-at-insert (not done in this PR)

A proper implementation would:

1. Add `padded_dim: usize` field to `HnswIndex` storing the aligned target
2. At `add()`/`add_batch()` time, pad each input vector once to `padded_dim`
3. At `search_with_ef()` time, pad the query once before dispatching
4. `UnifiedDistanceParams::compute()` sees only pre-aligned slices, zero
   per-call overhead

This changes the HNSW storage layout (wastes ~28 f32 per vector = 112 bytes
extra at 100D→128D) but eliminates the padding cost from the distance hot
path. Left as follow-up.

The v4 padding config **remains in the API** because it's correct (tests pass)
and because a future pad-at-insert implementation would still use the flag
as its "downstream consumer should see aligned inputs" signal.

## Padding Correctness Verification

Six new unit tests verify that zero-padding is semantically neutral for all
four metrics:

```
$ cargo test -p ruvector-core --lib --release -- advanced::eml::tests::test_padding
test test_padding_preserves_euclidean ... ok
test test_padding_preserves_cosine ... ok
test test_padding_preserves_dot_product ... ok
test test_padding_preserves_manhattan ... ok
test test_padding_noop_when_already_aligned ... ok
test test_padding_batch_compute ... ok

test result: ok. 6 passed; 0 failed
```

The padded result matches the unpadded result to within float epsilon
(1e-3 to 1e-6 depending on metric). So the *correctness* of padding is
established — the issue is purely the per-call implementation overhead.

## Final Recommendation (unchanged from v3 post-padding test)

1. **Ship all EML features as strict opt-in**, defaults unchanged. Done: this PR.
2. **Keep `pad_to_power_of_two` in the API** as a future hook. A pad-at-insert
   implementation can pair with it later without an API change.
3. **Document clearly** that current EML wins are reliable on power-of-two
   dimensions (SIFT1M: 128D, +14% QPS). Non-aligned dimensions may regress
   unless a user opts into the experimental padded path *and* their workload's
   per-call padding overhead is tolerable (e.g. batch scoring, not latency-
   critical HNSW).
4. **Close issue #351** with links to all four proof reports (v1 disproof →
   v2 SIMD win → v3 real-data mixed → v4 padding test) as the complete
   evidence chain.
